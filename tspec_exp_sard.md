
# T-Spec Expansion: Поддержка локализации (L10n) и системы логирования

## 1. Общие сведения (Front Matter)

* **Дополнение к версии:** 1.4
* **Модули:** Frontend (UI L10n), Backend (Dev-Logging)
* **Цель:** Обеспечить прозрачность процессов под капотом для разработчика и перевести пользовательский интерфейс на русский язык.

## 2. Архитектурные ограничения (Constraints)

* ❌ **ЗАПРЕЩЕНО (Frontend):** Использование тяжелых библиотек интернационализации (типа `i18next` или `vue-i18n`).
* ✅ **РАЗРЕШЕНО (Frontend):** Использование легковесного словаря (JSON/JS Object) и HTML-атрибутов `data-i18n` для Vanilla JS.
* ❌ **ЗАПРЕЩЕНО (Backend):** Вывод логов через простой `print()`.
* ✅ **РАЗРЕШЕНО (Backend):** Использование стандартной Python-библиотеки `logging` с разделением по уровням (DEBUG, INFO, ERROR).

## 3. Требования к локализации интерфейса (Frontend L10n)

Так как приложение нацелено на русскоязычного пользователя (вывод T-Spec и дебаты идут на русском), интерфейс должен быть переведен. Чтобы не хардкодить русский текст прямо в HTML (и сохранить возможность вернуть английский), реализуется механизм простого словаря.

**FR-L1: Словарь переводов (Translation Dictionary)**

* В файл `app.js` (или отдельный `i18n.js`) добавляется константа-словарь `const translations = { ru: {...}, en: {...} }`.
* Все статические тексты из `index.html` (кнопки, заголовки, плейсхолдеры) выносятся в ключи этого словаря.

**FR-L2: Разметка DOM-элементов**

* В `index.html` текстовым элементам добавляется атрибут `data-i18n="ключ_из_словаря"`.
* Пример: `<button onclick="uploadFile()" data-i18n="btn_upload">Upload Context File</button>`.

**FR-L3: Функция рендеринга перевода**

* При загрузке страницы (`DOMContentLoaded`) JS-функция проходит по всем элементам с атрибутом `data-i18n` и заменяет их `innerText` на соответствующее значение из словаря `ru`.
* Динамические сообщения в `app.js` (ошибки API, алерты "Please create or select a project first") также должны браться из этого словаря.

## 4. Требования к системе логирования (Backend Dev-Logging)

Логирование должно быть достаточно подробным, чтобы разработчик мог скопировать промпт, вставить его в AI Studio и воспроизвести ошибку.

**FR-LOG1: Инициализация логгера (Python `logging`)**

* В `app.py` настраивается глобальный логгер `sard_logger`.
* В режиме разработки (`app.debug = True` или переменная окружения `FLASK_ENV=development`) уровень логирования устанавливается на `logging.DEBUG`.
* Логи выводятся в консоль (StreamHandler) с подробным форматированием: `[%(asctime)s] %(levelname)s in %(module)s: %(message)s`.

**FR-LOG2: Точки трассировки (Trace Points)**
Логгер должен перехватывать следующие ключевые события:

1. **Инициализация стейта (INFO):** Создание нового проекта или смена `state_id`.
2. **Сборка контекста (DEBUG):** Перед отправкой запроса в Gemini бэкенд должен выводить в лог *полный текст* собранного промпта:
* `[DEBUG] AI Request Context: SystemPrompt_Length=..., ApprovedBlocks_Count=..., ChatHistory_Length=...`


3. **Запрос к Gemini API (INFO):** Фиксация старта запроса и выбранной модели (например, `gemini-3.1-pro-preview`).
4. **Ответ от Gemini API (DEBUG):** Вывод сырого ответа от модели до парсинга JSON:
* `[DEBUG] Raw AI Response: <сырой текст>`


5. **Ошибки и Fallback (ERROR/WARNING):** Логирование сбоев парсинга (`JSONDecodeError`), таймаутов и срабатывания механизма автоматического повтора (Retry).
6. **AI-Дистилляция (INFO):** Логирование процесса сжатия файла через модель `gemini-2.5-flash`.

## 5. План действий (Next Steps)

Для реализации этой спецификации потребуется внести изменения всего в два файла из твоего текущего репозитория:

1. В `app.py` импортировать `logging` и расставить вызовы `app.logger.debug()` в роутах `/api/chat` и `/api/upload`.
2. В `app.js` добавить объект словаря и функцию-обертку для перевода интерфейса, а в `index.html` добавить атрибуты `data-i18n`.
