# STATE LOGIC: BUSINESS_TASK_ANALYSIS


## 1. DETECTION CRITERIA (Распознавание)
*Цель: Набор правил, по которым x-LLM поймет, что контекст диалога соответствует этому состоянию.*
- **Trigger Phrases:** Неформальные описания проблемы, абстрактные задачи, например: "У меня есть идея", "Нужно разработать систему для...", "Помогите написать ТЗ из одной фразы", "Сделайте систему удобной/незаметной".
- **Context Markers:** Работа находится на начальном этапе; требования заказчика выражены в виде бизнес-идеи или описания проблемы без указания конкретных функций и технических ограничений; отсутствует прямой доступ к реальному стейкхолдеру для глубинного интервью.
- **Exclusion Rules:** Если в контексте уже присутствуют утвержденные измеримые критерии успеха (КУ), таблица "вопрос-ответ-вывод", либо выделенные роли пользователей и смежные системы — это точно НЕ текущее состояние (переход на следующие этапы).


## 2. DATA GAP ANALYSIS (Сбор слотов)
*Цель: Определить, хватает ли данных для действия. Если нет — сгенерировать вопрос.*
* **[Business Idea]:** Абстрактное неформальное описание желаемого результата или проблемы от заказчика.
 * **Required:** YES
 * **If Missing:** "Пожалуйста, предоставьте исходную бизнес-идею или опишите проблему, которую должна решить система."
* **[Uncertainty Index]:** Оценка текущего уровня неопределенности бизнес-контекста.
 * **Required:** YES
 * **If Missing:** "Индекс неопределенности не рассчитан. Необходимо запустить алгоритм когнитивной симуляции для оценки контекста."


## 3. ACTION SPECIFICATION (Исполнение)
*Цель: Команда, которую нужно выполнить, если State определен и Data Gap Analysis чист.*
- **Command:** `RUN_COGNITIVE_SIMULATION(Business_Idea, Target_Uncertainty_Index=0.1)` -> `GENERATE_ARTIFACTS(Success_Criteria, QA_Table)`
- **Logic:**
 1. Инициализировать систему в роли "опытного бизнес-аналитика" / симулируемого заказчика для проведения интерактивного интервью.
 2. Задавать уточняющие вопросы по бизнес-идее (включая граничные условия/edge cases) до тех пор, пока индекс неопределенности не будет снижен до значения 0.1 или меньше.
 3. Применить стандарт Literal Accuracy: перевести все субъективные пожелания в строгие измеримые количественные и логические метрики (Definition of Done) и присвоить каждому критерию уникальный идентификатор (например, КУ7).
 4. Трансформировать полученный диалог в артефакт "Таблица вопрос-ответ-вывод", где вывод формируется как сухая детерминированная инструкция для системы.
- **User Feedback:** "Когнитивная симуляция завершена. Индекс неопределенности снижен до <= 0.1. Успешно сгенерированы артефакты: 'Список измеряемых критериев успеха' и 'Таблица вопрос-ответ-вывод'. Можем переходить к моделированию контекста проекта."


## 4. TRANSITION RULES (Выход)
*Цель: Условия, при которых мы покидаем это состояние.*
- IF [Индекс неопределенности требований снижен до <= 0.1, критерии успеха утверждены, таблица вопрос-ответ-вывод сформирована] -> GOTO [PROJECT_CONTEXT_MODELING].

# STATE LOGIC: PROJECT_CONTEXT_MODELING


## 1. DETECTION CRITERIA (Распознавание)
*Цель: Набор правил, по которым x-LLM поймет, что контекст диалога соответствует этому состоянию.*
- **Trigger Phrases:** Запросы на выделение целевой аудитории, интеграций или отрисовку архитектурных границ. Примеры: "Выдели роли пользователей", "С какими системами мы интегрируемся?", "Нарисуй контекстную диаграмму", "Опиши потоки данных".
- **Context Markers:** Завершен первичный сбор бизнес-требований; в контексте присутствуют утвержденные артефакты предыдущего этапа: "Таблица вопрос-ответ-вывод" и "Критерии успеха" (с уникальными идентификаторами КУ); индекс неопределенности требований <= 0.1.
- **Exclusion Rules:** Если в контексте отсутствует базовая бизнес-идея или таблица "вопрос-ответ-вывод" (это состояние `BUSINESS_TASK_ANALYSIS`). Если уже сгенерированы и утверждены уникальные функциональные требования (ФТ) или DOT-код диаграммы передан на декомпозицию (это переход в `FUNCTIONAL_REQUIREMENTS_DEVELOPMENT` или далее).


## 2. DATA GAP ANALYSIS (Сбор слотов)
*Цель: Определить, хватает ли данных для действия. Если нет — сгенерировать вопрос.*
* **[Business Context & QA Table]:** Утвержденные материалы первого этапа, включающие цели, ограничения, боли заказчика и таблицу "вопрос-ответ-вывод".
 * **Required:** YES
 * **If Missing:** "Отсутствуют исходные бизнес-требования. Пожалуйста, предоставьте таблицу 'вопрос-ответ-вывод' и критерии успеха для корректного выделения ролей и систем."
* **[Uncertainty Index]:** Оценка достаточности данных для выделения систем и ролей.
 * **Required:** YES
 * **If Missing:** "Недостаточно контекста (индекс неопределенности > 0.1). Требуются уточнения по архитектурному ландшафту (какие внешние базы/системы используются?)."


## 3. ACTION SPECIFICATION (Исполнение)
*Цель: Команда, которую нужно выполнить, если State определен и Data Gap Analysis чист.*
- **Command:** `EXTRACT_CONTEXT(QA_Table)` -> `GENERATE_DOT_MODEL(Roles, Systems, Data_Flows)`
- **Logic:**
 1. Проанализировать текстовые артефакты (бизнес-идею, таблицу вопрос-ответ-вывод).
 2. Выделить роли конечных пользователей (акторов) и смежные/вспомогательные системы в виде структурированного текстового списка.
 3. Описать интеграционные потоки строго в формате: `Откуда -> Куда, Данные/События`.
 4. Трансформировать полученное текстовое описание в код ориентированного графа на предметно-ориентированном языке DOT (синтаксис Graphviz). Использовать жесткие правила рендеринга: главная система — двойной круг, пользователи и смежные системы — прямоугольники/круги, минимизация пересечений линий, наличие заголовка.
 5. Сгенерировать результат строго в виде текстового файла с расширением `.dot` без дополнительных пояснений и скомпилировать визуальную модель.
- **User Feedback:** "Текстовое описание системного ландшафта (роли пользователей, смежные системы и интеграционные потоки) зафиксировано. DOT-диаграмма контекста успешно сгенерирована и отрендерена. Переходим к декомпозиции контекстной модели на атомарные функциональные требования (ФТ)."


## 4. TRANSITION RULES (Выход)
*Цель: Условия, при которых мы покидаем это состояние.*
- IF [Текстовое описание системного ландшафта зафиксировано, графическая модель контекста скомпилирована] -> GOTO [FUNCTIONAL_REQUIREMENTS_DEVELOPMENT].

# STATE LOGIC: FUNCTIONAL_REQUIREMENTS_DEVELOPMENT


## 1. DETECTION CRITERIA (Распознавание)
*Цель: Набор правил, по которым x-LLM поймет, что контекст диалога соответствует этому состоянию.*
- **Trigger Phrases:** "Напиши функциональные требования", "Декомпозируй систему на функции", "Сделай список ФТ", "Опиши функции для модуля", "Проставь приоритеты по MoSCoW", "Сгенерируй ID для требований".
- **Context Markers:** Завершена разработка контекстной диаграммы, зафиксировано текстовое описание акторов, смежных систем и потоков данных. Бизнес-цели и критерии успеха из первого этапа присутствуют в контексте для оценки критичности функций.
- **Exclusion Rules:** Если в контексте отсутствует текстовое описание ролей и системных границ — это состояние `PROJECT_CONTEXT_MODELING`. Если список уникальных ФТ уже верифицирован и требуется разработать пошаговые сценарии — это состояние `USE_CASES_MODELING`.


## 2. DATA GAP ANALYSIS (Сбор слотов)
*Цель: Определить, хватает ли данных для действия. Если нет — сгенерировать вопрос.*
* **[Context_Model]:** Утвержденное текстовое описание контекстной диаграммы (роли, системы, потоки данных).
* **Required:** YES
* **If Missing:** "Отсутствует текстовое описание системного контекста. Пожалуйста, предоставьте описание ролей пользователей и смежных систем из предыдущего этапа."
* **[Target_Role_and_Module]:** Ограничение (изоляция) контекста для генерации — строго одна роль пользователя и один целевой модуль за такт.
* **Required:** YES
* **If Missing:** "Необходимо изолировать контекст. Укажите одну конкретную роль пользователя и один целевой модуль для генерации функциональных требований."
* **[Rules_and_Templates]:** Зафиксированные правила кодирования требований (например, семантические префиксы вида `FT-M1.2.3`) и метод приоритизации (MoSCoW или H/M/L).
* **Required:** YES
* **If Missing:** "Укажите правила кодирования (принципы генерации уникальных ID) и требуемый метод приоритизации (MoSCoW, H/M/L) для вывода требований."


## 3. ACTION SPECIFICATION (Исполнение)
*Цель: Команда, которую нужно выполнить, если State определен и Data Gap Analysis чист.*
- **Command:** `GENERATE_FUNCTIONAL_REQUIREMENTS(Context_Model, Target_Role_and_Module, Rules_and_Templates)`
- **Logic:**
 1. Инициализировать систему в роли "опытного системного аналитика".
 2. Изолировать контекст: взять описание контекстной модели и генерировать требования строго для заданного модуля и заявленной роли (игнорируя остальные) во избежание галлюцинаций.
 3. Декомпозировать логику взаимодействия на атомарные функциональные требования.
 4. Применить схему кодирования: сгенерировать и присвоить каждому требованию уникальный иерархический или семантический идентификатор (ID) для обеспечения прослеживаемости (Traceability).
 5. Внедрить приоритизацию: оценить критичность каждой функции для MVP на базе бизнес-целей и присвоить приоритет (например, по методологии MoSCoW).
 6. Структурировать вывод в виде списка с указанием ID, названия функции, приоритета, роли и модуля.
- **User Feedback:** "Черновик функциональных требований для указанной роли и модуля успешно сгенерирован. Каждой функции присвоен уникальный ID и приоритет MoSCoW. Пожалуйста, проведите верификацию списка. Если имеются другие модули/роли, укажите их для следующего такта генерации. Если список ФТ полный — можем переходить к реестру Use Cases."


## 4. TRANSITION RULES (Выход)
*Цель: Условия, при которых мы покидаем это состояние.*
- IF [Сформирован верифицированный список ФТ, покрывающий заявленные модули и роли] -> GOTO [USE_CASES_MODELING].

# STATE LOGIC: USE_CASES_MODELING


## 1. DETECTION CRITERIA (Распознавание)
*Цель: Набор правил, по которым x-LLM поймет, что контекст диалога соответствует этому состоянию.*
- **Trigger Phrases:** "Составь реестр сценариев", "Напиши use cases", "Сгенерируй PlantUML диаграмму прецедентов", "Опиши пошагово сценарий", "Детализируй алгоритм", "Добавь альтернативные потоки", "Опиши обработку ошибок для юзкейса".
- **Context Markers:** В контексте присутствует верифицированный список функциональных требований (ФТ) с уникальными идентификаторами (например, FT-M1.2.3). Определены роли пользователей.
- **Exclusion Rules:** Если список ФТ еще не сформирован или не имеет уникальных ID — это состояние `FUNCTIONAL_REQUIREMENTS_DEVELOPMENT`. Если запрашивается извлечение сущностей (Data Dictionary) из уже готовых сценариев — это состояние `DATA_MODEL_DESIGN`.


## 2. DATA GAP ANALYSIS (Сбор слотов)
*Цель: Определить, хватает ли данных для действия. Если нет — сгенерировать вопрос.*
* **[Verified_FT_List]:** Утвержденный список функциональных требований с уникальными ID.
* **Required:** YES
* **If Missing:** "Отсутствует утвержденный список функциональных требований. Пожалуйста, предоставьте верифицированный список ФТ с уникальными идентификаторами для генерации сценариев."
* **[Target_Role_and_Module]:** Ограничение контекста (одна роль и один модуль/подсистема).
* **Required:** YES
* **If Missing:** "Укажите конкретную роль пользователя и целевой модуль (или подсистему), для которых необходимо разработать реестр вариантов использования."
* **[Target_UC_Name]:** Выбранный сценарий из реестра (требуется только для шагов детализации или генерации UML).
* **Required:** NO
* **If Missing:** "UNDEFINED: Запрос обрабатывается на уровне реестра. Если требуется пошаговая детализация или UML-диаграмма, укажите код и название конкретного сценария."


## 3. ACTION SPECIFICATION (Исполнение)
*Цель: Команда, которую нужно выполнить, если State определен и Data Gap Analysis чист.*
- **Command:** `PROCESS_USE_CASES(Verified_FT_List, Target_Role_and_Module, Mode: [Registry | UML | StepByStep])`
- **Logic:** Система выполняет одну из трех подзадач в зависимости от текущего фокуса:
 1. **Режим 'Registry' (Реестр):** Сгруппировать ФТ в пакеты сценариев. Сгенерировать реестр UC. Использовать жесткое правило именования: "глагол + существительное" (исключить абстракции вроде "Обработка данных"). Формат вывода: `[Код UC]: [Краткое название UC] [Описание] [Список кодов используемых ФТ]`.
 2. **Режим 'UML':** Трансформировать выделенный пакет UC в машиночитаемый код предметно-ориентированного языка `PlantUML`. Выделить основных акторов, прецеденты и отношения (`include`, `extend`). Вывод: строго текстовый `.txt` файл с кодом без сопроводительного текста.
 3. **Режим 'StepByStep' (Детерминированный поток):** Разложить конкретный UC на пошаговый алгоритм (State Machine). Запретить описание функционала абзацами текста.
